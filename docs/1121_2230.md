后端需求分析报告
基于对 frontend 文件夹内前端代码（特别是 
src/data/mockData.ts
, 
App.tsx
 及各组件）的分析，为了支持现有的前端功能，后端需要提供以下功能模块和 API 接口。

1. 核心数据模型 (Data Models)
后端数据库需要存储以下核心实体：

1.1 用户 (User)
基础信息: ID, 用户名, 邮箱, 头像 URL, 昵称。
科研画像 (Memory):
关注点 (Focus): 领域 (Domains), 关键词 (Keywords), 关注作者 (Authors), 关注机构 (Institutions)。
上下文 (Context): 当前任务 (Current Task), 未来目标 (Future Goal), 阶段 (Stage), 目的 (Purpose), 学习模式 (Learning Mode - 基础/创新)。
设置: 偏好设置 (如每日推送时间等)。
1.2 论文 (Paper)
元数据: ID (ArXiv ID), 标题, 作者列表, 类别 (Category), 发布日期, 摘要, 链接 (PDF, ArXiv, HTML)。
AI 分析数据:
TL;DR: 一句话总结。
建议 (Suggestion): 精读/泛读/略读。
标签 (Tags): AI 生成的标签。
详细分析 (Details): 动机 (Motivation), 方法 (Method), 结果 (Result), 结论 (Conclusion)。
推荐理由 (Why This Paper): 结合用户个性化信息与论文摘要生成的推荐理由 (避免上下文过长)。
1.3 报告 (Report)
基础信息: ID, 标题, 日期, 摘要。
内容: 细粒度文本结构，要求每一句或极短的片段都能关联到引用的论文 ID。生成算法需支持这种细粒度的引用标注。
1.4 用户交互与记忆更新 (Interactions & Memory)
交互数据: 点赞/不喜欢 (isLiked, dislikeReason), 阅读记录。
记忆更新: 需要一个机制，根据用户的交互行为（如不喜欢某篇论文的原因）动态更新用户的科研画像 (Memory)。
数据库要求: 需要一个健壮、可维护且支持多用户并发更新的数据库方案。

3. 后台任务与 AI 能力 (Background Jobs & AI)
除了直接响应前端请求，后端还需要具备以下后台处理能力：

3.1 数据爬取 (Crawler)
每日定时任务: 自动从 ArXiv 爬取指定领域 (如 cs.AI, cs.LG) 的最新论文。
3.2 AI 处理流水线 (AI Pipeline)
论文筛选: 根据用户设定的关键词和领域，初步筛选相关论文。
内容分析 (LLM):
生成中文 TL;DR 和摘要。
提取动机、方法、结果、结论。
打标签 (Tags)。
生成阅读建议 (精读/泛读)。
个性化推荐:
基于用户画像和论文摘要生成 "Why This Paper" 推荐理由。
报告生成:
基于每日/每周的高分论文，自动生成综述报告。
关键要求: 生成内容需精确到句子的引用标注。
3.3 向量检索 (RAG - 可选但推荐)
为了支持更精准的推荐和“类似论文”查找，建议建立论文的向量数据库 (Vector DB)，支持语义检索。
4. 认证与安全 (Authentication)
需要实现用户注册/登录系统 (JWT 或 Session)。
所有 /api/user/* 和写操作接口都需要鉴权。
5. 总结
前端展示了高度个性化和智能化的功能，这意味着后端不仅仅是一个简单的 CRUD 系统，而是一个集成了 爬虫 -> AI 分析 -> 个性化推荐 -> 知识管理 的复杂系统。核心难点在于 AI 流水线的稳定性和推荐算法的准确性。
